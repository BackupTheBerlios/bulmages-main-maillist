> El tema radica en la codificación de ls UTF8 y UTF16  (cuyos carácteres

Espero que me perdones si soy demasiado pesado, pero siento la necesidad de  
insistir en este tema. El ejemplo siguiente es un trozo de código tomado de 
la propia biblioteca Qt4.5, QUtf32Codec::convertFromUnicode()

00496         for (int i = 0; i < len; ++i) {
00497             uint cp = uc[i].unicode();
00498             if (uc[i].isHighSurrogate() && i < len - 1)
00499                 cp = QChar::surrogateToUcs4(cp, uc[++i].unicode());

De este código deduzco un par de cosas: 1.-cuando encontramos un surrogate 
pair, primero viene el high y luego el low, 2.-antes de acceder al siguiente 
carácter, mejor comprobar que no se ha alcanzado el límite, en este caso un 
array de QChar. Hay varios otros ejemplos en Qt. 

Si miras como está definido QChar::isHighSurrogate(), en qchar.h:276, es una 
simple comprobación de rango. 

00276     inline bool isHighSurrogate() const {
00277         return ((ucs & 0xfc00) == 0xd800);
00278     }
00279     inline bool isLowSurrogate() const {
00280         return ((ucs & 0xfc00) == 0xdc00);
00281     }
 
Además de lo anterior, BlDbRecord::generaRML() está tratando datos que se leen 
desde un fichero y se tratan por sustitución de tags. Obviamente, muchas 
cosas pueden ir mal, como por ejemplo corrupción de los datos haciendo que 
aparezca un HighSurrogate donde no debería. Y como siempre en este oficio, 
aquí se puede aplicar la ley de Murphy, y prever cualquiera de sus corolarios 
(http://es.wikipedia.org/wiki/Ley_de_Murphy).
