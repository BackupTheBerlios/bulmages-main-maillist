Estoy comprobando la compilación en en mi wheezy de amd64 y he tenido
que cambiar una pequeña cosa para que se compilen en esta arquitectura.

*   En debian/control cambia "Architecture" del Package:bulmages-release

         i386  -->  any

    de otra los paquetes serían ignorados por arquitecturas
    distintas a i386 (incluída amd64).

Con este cambio he conseguido compilar el paquete en amd64.

Efectivamente, los paquetes que están en mi web mtelleria.be son una
"exportación" de los originales que están en las ramas de repositorio
git de gitorious.

Toda la "nomenclatura" se basa en información de git la cual no está
disponible en los paquetes de la web en sí.  Para obtenerla es
necesario obtener el paquete fuente desde git con:

   git clone git://gitorious.org/bulmages/bulmages.git

de hecho si vas a modificar cosas es mejor que te hagas una cuenta de
gitorious si no la tienes ya y te clees un clon de dicho repositorio.

Si no estas familiarizado con git, te recomiendo que te hagas una serie
de tutoriales y lo empieces a usar en local (estrictamente no hace
falta un servidor git) y con gitorious como repositorio remoto.

> Por otra parte, las tags (etiquetas) a que te refieres en la wiki del

> (wiki<http://redmine.bulmages.es/wiki/bulmages-main/Bulmages_y_git-buildpackage/diff?version=13>)

No, exactamente.  Cuando hablo de tag hablo de una tag de git, que como
todas las tags en git, apuntan a commits.

NOTA: Dada la gran cantidad de tags que se generan en el empaquetado,
      las he puesto en mi clone de bulmages mtelleria-bulmages.

https://gitorious.org/~mtelleria/bulmages/mtelleria-bulmages

Dichas tags (y sus commits) sirven para referenciar el contenido del
orig.tar.gz y el workspace del empaquetado (orig.tar.gz + directorio
debian).

Por ejemplo refiriéndonos a los valores 1fa1cab1 y ef7f8f75:

*  1fa1cab1 y ef7f8f75 son commits de git en las ramas release y master
   respectivamente.  El contenido de dichos commits es lo que
   constituye el orig.tar.gz.

*  Las tags correspondientes a 1fa1cab1 serian:

   -  Para el commit usado como orig.tar.gz:
      - release/0.14.0-1fa1cab1

   -  Para los commits usados como workspace de empaquetado tenemos
      varios valores dependiendo de la distribución y de la versión

      - release-squeeze/0.14.0-1fa1cab1-squeeze1
      - release-debian/0.14.0-1fa1cab1-sid1
      - release-debian/0.14.0-1fa1cab1-sid2

      cada tag apunta al commit correspondiente a la versión del
      paquete.  De hecho lo que está después de / ha de coincidir con
      la versión puesta en debian/changelog.

*  Idem para ef7f8f75 pero esta vez en la rama master, sólo que aquí
   ponemos además el timestamp del commit git en GMT.

   -  Commit usado como orig.tar.gz
      - master/0.14.0.20130111.1234-ef7f8f75

   -  Commits usados como workspace de empaquetado.
      - master-debian/0.14.0.20130111.1234-ef7f8f75-1 
          (en este se me olvidó añadir lo de sid)
      - master-debian/0.14.0.20130111.1234-ef7f8f75-sid2
      - master-debian/0.14.0.20130111.1234-ef7f8f75-sid3

      - master-squeeze/0.14.0.20130111.1234-ef7f8f75-1
      - master-squeeze/0.14.0.20130111.1234-ef7f8f75-squeeze1
      - master-squeeze/0.14.0.20130111.1234-ef7f8f75-squeeze2
