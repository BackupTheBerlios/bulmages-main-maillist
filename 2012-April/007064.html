> Yo lo mantendria en la misma rama. Como bien dices, en git las ramas son
> un poco diferentes, y generalmente estan pensadas para ir desarrollando
> funcionalidades manteniendo un codigo base mas o menos estable. Desde mi
> punto de vista es el mismo caso que puede tener, por ejemplo, un paquete
> debian, que basicamente varia con respecto al resto del programa en un
> directorio. En ese caso se suele crear una rama para desarrollar una
> funcionalidad (crear un paquete debian a partir del codigo fuente) y,
> una vez que se termina de desarrollar la funcionalidad se mergea con la
> rama correspondiente. Asi no tienes que tener una rama para crear paquetes
> debian que tengas que actualizar cada vez que quieras crear un paquete.
>
> Otra cosa es que el codigo en windows compartiera poco con el resto, con
> lo que si podria tener sentido tenerlo en una rama aparte (si ves mi
> repositorio en gitorious he hecho eso para la documentacion o la web, ya
> que quien esta trabajando en la documentacion o en la web no le preocupa
> si alguien ha hecho un cambio en el codigo fuente). E incluso para estos
> casos mucha gente prefiere tener repositorios aparte.
>
> Por cierto, en git los tags son muy utiles, porque te permiten tener una
> unica rama estable (no una por version), incluir una tag cada vez que se
> lance una version y, en caso de que se encuentre un bug en una version
> estable que no sea la ultima git te permite crear branches a partir de
> una tag, con lo cual no te obliga a tener una branch por cada version
> estable que haya.
